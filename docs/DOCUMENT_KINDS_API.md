# Document-kinds routing — API contract

**What it answers.** "I need document kind *K* (Registerauszug / Gesellschafterliste /
Gesellschaftsvertrag) for a company in jurisdiction *J* (legal form *F*) — what do I do?"
The consumer never names a vendor; the aggregator resolves everything.

This is the **actionable** layer over the cited **evidence** layer. Keep the two separate.

## Two layers (do not merge them)

1. **Registry truth — `curation/document_kinds_evidence.json`** (vendor-neutral). Per
   `(jurisdiction × legalForm × kind)`: which document the official registry *evidences*, with
   statute citations and verbatim quotes. This is about reality; it does not know who sources it.
2. **Routing — `curation/document_kinds_routing.json`** (generated by `scripts/build_routing.py`).
   Turns each evidence row into a decision and resolves the **vendor**. This is the API's backing
   data. **We are a vendor-neutral aggregator** (KYC.com, handelsregister.de, …); which vendor
   serves a jurisdiction lives *only* here. Adding or swapping a vendor must not touch layer 1 and
   must not change the consumer contract.

The routing table is regenerated, not hand-edited: change the evidence file or the small maps in
`build_routing.py`, then re-run it. The backend just serves lookups over the output.

## Endpoint

```
GET /restapi/v1/document-kinds/route?jurisdiction={ISO2}&kind={KIND}&legalForm={form}
```

- `jurisdiction` (required) — ISO 3166-1 alpha-2 (`GB`, `IT`, `AT`, …).
- `kind` (required) — `REGISTERAUSZUG | GESELLSCHAFTERLISTE | GESELLSCHAFTSVERTRAG`.
- `legalForm` (optional) — the resolved company form (`GmbH`, `Ltd`, `Srl`, …). **If omitted**,
  the answer can differ by form (AT GmbH vs AG), so the service returns **all matching forms**
  (an array) rather than guessing. Provide the form to get a single decision.

## Response (one route)

```json
{
  "jurisdiction": "GB",
  "legalForm": "Ltd",
  "kind": "GESELLSCHAFTERLISTE",
  "availability": "off_registry",
  "action": "manual",
  "orderable": false,
  "completeness": null,
  "document": null,
  "vendor": "kyc.com",
  "fallback": "company_register",
  "confidence": "high",
  "needsReview": false,
  "sources": [ { "basis": "Companies Act 2006, s.853F(4)", "url": "https://…" } ]
}
```

For an orderable kind (e.g. `MT / Ltd / GESELLSCHAFTERLISTE`):

```json
{
  "availability": "additional",
  "action": "order_with_warning",
  "orderable": true,
  "completeness": "snapshot",
  "flags": ["snapshot"],
  "order": {
    "vendor": "kyc.com",
    "model": "case_level",
    "registryName": "annual return (Seventh Schedule, Part 3 — List of Past and Present Members)",
    "howToObtain": "additional_within_case",
    "vendorCatalog": "jurisdiction_matrix.json"
  },
  "vendor": "kyc.com",
  "fallback": null,
  "confidence": "high"
}
```

### Field semantics

| field | meaning |
|---|---|
| `availability` | `base` (comes with the case) · `additional` (extra order) · `off_registry` (registry doesn't evidence it, but a company-held route exists) · `none` (no route) |
| `action` | the enum the UI branches on — see below |
| `orderable` | true for `base`/`additional` |
| `completeness` | `full` · `delta` · `threshold` · `snapshot` · `sole_only` · `founders_only` · `conditional` — **why** a provable document is less than a full current list. Drives the warning badge. |
| `flags` | cross-cutting concerns that don't affect completeness — currently `nominee` (registered ≠ beneficial holder). See below. |
| `order` | how to obtain it via the resolved vendor (`model`, `howToObtain`, `registryName`, `vendorCatalog`). See Vendor ordering model. |
| `vendor` | resolved aggregation vendor. Consumer displays nothing vendor-specific; it's here for the order call and audit |
| `dataBackup` | (shareholder-list kind) KYC.com's parsed shareholder **data** for the jurisdiction — a backup when the proof document is unavailable. `{available, fields[], isProof:false}`. See below. |
| `fallback` | `company_register` · `ubo_register` · `none` — where to send the user when `action = manual` |
| `needsReview` | true = our completeness mapping and the evidence caveats disagree; do not treat as authoritative until curated |
| `sources` | citations carried from the evidence layer for audit / display |

### `action` vocabulary — what the app does

| action | UI behaviour |
|---|---|
| `use_delivered` | Document is in the case bundle. Pick it from the delivered files (classify direction). |
| `use_delivered_with_warning` | As above, plus show the completeness badge. |
| `order` | Place an explicit additional order via `vendor`. |
| `order_with_warning` | As above, plus surface the completeness badge (e.g. "annual snapshot, up to ~13 months old"). |
| `manual` | No registry document. Show the `fallback` (e.g. "obtain the company's own register of members"). No order button. |
| `unavailable` | No route in this jurisdiction. Disable, show the reason. |

This is why the contract is safe: England/Gesellschafterliste returns `manual`, so the app
**cannot** hand a customer a CS01 labelled "shareholder list" — the decision refuses to offer it.

## Vendor neutrality (the core requirement)

- The consumer contract has **no vendor input and no vendor-specific branching**. It asks for a
  *kind in a jurisdiction*; the aggregator picks the vendor.
- `vendorRouting` in the routing file maps jurisdiction → vendor. Today: **`DE → handelsregister.de`**,
  everything else `→ kyc.com`. Extend per jurisdiction as vendors are added (each jurisdiction may
  have its own vendor).
- Swapping a jurisdiction's vendor = edit `vendorRouting` and re-map `vendorOrderType`. Nothing
  else changes — not the evidence, not the consumer contract.

## Vendor ordering model — granularity differs per vendor

There is **no per-document SKU to hunt for**. Ordering granularity is a property of the vendor,
and bridging it to a uniform document-level request is our aggregation layer's job:

- **KYC.com — case-level.** You order a *case* for a company; base documents come with it,
  "additional" documents are requestable *within* the case (extra charge). The
  `jurisdiction_matrix.json` base/additional lists **are** the KYC.com catalog — that JSON is all
  there is, because you order the case, not the document.
- **handelsregister.de — document-level.** You order the specific German document directly.

Each route carries an `order` object reflecting this:

```json
"order": {
  "vendor": "kyc.com",
  "model": "case_level",
  "registryName": "annual return (Seventh Schedule, Part 3 …)",
  "howToObtain": "additional_within_case",   // or "included_in_case" (base) / "order_document"
  "vendorCatalog": "jurisdiction_matrix.json",
  "aggregationNote": "Document-level request provided by our aggregation layer; for a case-level vendor no per-document SKU exists."
}
```

So the consumer always issues a **document-level** request; our aggregation layer maps it down to
whatever the vendor supports — a case order (KYC.com) or a document order (handelsregister.de). The
integration work is that mapping in our layer, not a missing vendor SKU. `howToObtain` tells the
order code which path to take.

## Data backup — deliver the data when the document can't be sourced

The proof rule ("a document is the proof, data is not a substitute") governs *completeness* and
*evidential weight*. But where the proof document **cannot** be sourced at all, delivering the
available data — clearly labelled as data, not proof — is a real fallback the operator should be
offered. For the shareholder-list kind, KYC.com's parsed `shareholders` fields (from
`jurisdiction_matrix.json`) are exactly that.

```json
"dataBackup": {
  "available": true,
  "fields": ["Name", "Share count", "Role", "Address (if available)"],
  "source": "kyc.com structured data (parsed)",
  "isProof": false,
  "note": "Data, not a proof document. Deliverable as a data sheet / generated document where the proof document is unavailable."
}
```

- **46 of 61** shareholder-list rows have this data; for **21 of them there is no proof document
  at all** — the data is the *only* route (England is one: `action:"manual"` but data available).
- The UI should present it as a **secondary** option, never as the shareholder list itself:
  "No proof document available — deliver KYC.com shareholder data as backup (data, not proof)."
- `isProof:false` is deliberate and must survive to the customer-facing label. A generated
  BetterCo data sheet is a convenience, not a registry document.
- Where `dataBackup.available` is false **and** the document is not provable (ES, SE, FI, NO, GG
  for the shareholder list), there is genuinely nothing — say so plainly.

## Cross-cutting flags (separate from completeness)

`flags` carries concerns that don't make the list *incomplete* but the operator must see —
currently `nominee` (the registered holder may be a nominee, not the beneficial owner; common in
CY/JE). Completeness answers "is it the full current list"; flags answer "what else to know about
the holders shown." Keep them distinct — a complete list of nominees is still complete.
